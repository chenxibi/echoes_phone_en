# 完整阅读器 - 拒绝片面理解

**核心原则：先完整阅读，再下结论。没有完整阅读的分析都是耍流氓。**

---

## 问题背景

用户经常遇到这样的问题：
- AI 只读了 README 就开始分析项目
- AI 读取了部分文件就给出片面结论
- AI 跳过关键配置文件导致理解错误
- AI 没有理解模块依赖关系就下判断
- AI 遗漏子目录内容导致分析不完整
- AI 没有交叉验证数据导致错误结论

**本 Skill 强制要求：在输出任何分析之前，必须完成完整阅读流程，并通过验证清单。**

---

## 🚨 强制阅读协议 [完整阅读生效 📚]

当用户要求分析/理解某个路径时，**必须**按以下步骤执行，**不可跳过、不可简化、不可假设**。

### Step 1: 全面扫描（必须首先执行）

**禁止行为**：
- ❌ 直接读取某个文件而不先了解整体结构
- ❌ 只扫描根目录，忽略子目录
- ❌ 假设"应该没有更多内容了"

**必须执行**：
```
1. 使用 list_files_recursive 获取完整目录结构
2. 统计文件总数、目录层级、子目录数量
3. 识别项目类型（通过文件扩展名和配置文件）
4. 标记所有关键文件位置
5. 记录所有子目录及其用途
```

**输出要求**：
```
[扫描完成] 项目结构概览
├── 总文件数: N
├── 目录层级: M
├── 子目录数: K
├── 项目类型: [Web/CLI/库/服务/未知]
├── 关键文件: [列出发现的配置文件、入口文件等]
├── 核心目录: [src/, lib/, app/ 等]
└── 子目录列表: [完整列出所有子目录]
```

### Step 2: 关键文件必读清单

以下文件类型**如存在则必须读取**，按优先级顺序：

**P0 - 项目定义文件（必须读）**：
- README.md / readme.md / README / SKILL.md
- package.json / Cargo.toml / go.mod / requirements.txt / pyproject.toml / pom.xml / build.gradle
- tsconfig.json / jsconfig.json
- manifest.yaml / codemaker.yaml / .codemakerrc
- .gitignore（了解项目边界）

**P1 - 入口与核心（必须读）**：
- 入口文件：main.* / index.* / app.* / __main__.py / cli.*
- 核心配置：config.* / settings.* / .env.example
- 路由/接口定义：routes.* / api.* / controllers.*

**P2 - 架构与类型（尽量读）**：
- 类型定义：*.d.ts / types.* / models.* / schemas.*
- 接口定义：*.proto / *.graphql / openapi.yaml
- 架构文档：ARCHITECTURE.md / docs/architecture.* / DESIGN.md

**P3 - 测试与示例（辅助理解）**：
- 测试文件：*.test.* / *.spec.* / tests/
- 示例代码：examples/ / demo.* / samples/

**P4 - 子目录深度内容（必须检查）**：
- 每个子目录下的 README 或说明文件
- 子目录特有的配置文件
- 子目录的入口/核心文件

### Step 3: 依赖关系梳理与递归读取

读取核心文件后，**必须**：

1. **识别导入依赖**：列出主要文件的 import/require 依赖
2. **递归读取关键依赖**：递归读取核心模块依赖的其他模块（至少 2-3 层）
3. **理解调用链**：
   - 入口 → 路由 → 控制器 → 服务 → 数据层
   - 或者：CLI → 命令解析 → 执行逻辑 → 输出
4. **检查循环依赖**：识别并记录模块间的循环依赖

**输出要求**：
```
[依赖梳理] 核心调用链
├── 入口: [文件路径]
├── 依赖模块: [列出关键依赖]
├── 数据流向: [描述主要数据流]
├── 递归深度: [读取的依赖层级]
└── 未读取文件: [如有，明确说明原因]
```

### Step 4: 交叉验证（新增）

**必须执行**：
1. **数据一致性检查**：对比不同文件中的相同数据是否一致
2. **版本号验证**：检查 package.json、manifest.yaml、README 中的版本是否一致
3. **引用验证**：验证文件中引用的其他文件是否真实存在
4. **配置验证**：检查配置文件之间的依赖关系是否正确

**输出要求**：
```
[交叉验证] 一致性检查
├── 版本一致性: [通过/失败 - 详情]
├── 数据一致性: [通过/失败 - 详情]
├── 引用有效性: [通过/失败 - 详情]
└── 发现问题: [列出发现的不一致或错误]
```

### Step 5: 深度补全（按需）

根据用户具体问题，**必须**深入读取：

- **分析功能** → 读取该功能的完整实现链
- **分析 Bug** → 读取错误位置 ±50 行上下文
- **分析性能** → 读取热点代码和相关配置
- **分析架构** → 读取核心模块间的交互代码
- **对比文档** → 读取所有相关文档进行交叉验证

## 🔍 错误指出与修正要求（新增）

在分析过程中，**必须**主动发现并指出以下问题：

### 必须指出的问题类型
1. **文档错误**：README、文档中的错误描述
2. **数据不一致**：同一数据在不同地方定义不同
3. **版本不匹配**：文件版本与实际功能不匹配
4. **引用错误**：引用的文件不存在或路径错误
5. **逻辑矛盾**：代码或文档中的逻辑矛盾
6. **遗漏内容**：明显缺失的关键内容

### 输出格式
```markdown
## 🚨 发现的问题

### 严重问题（必须修复）
1. **[问题类型]**: [问题描述]
   - 位置: [文件路径:行号]
   - 影响: [影响范围]
   - 建议: [具体修复方案]

### 警告问题（建议修复）
1. **[问题类型]**: [问题描述]
   - 位置: [文件路径:行号]
   - 建议: [改进建议]

### 建议优化
1. **[优化类型]**: [优化建议]
```

## 📏 项目规模适配策略

### 小型项目 (< 20 个文件)
**策略**：全量阅读
- 递归列出所有文件
- 读取所有代码文件和配置
- 执行完整交叉验证
- 理解完整项目

### 中型项目 (20-100 个文件)
**策略**：分层深入 + 强制子目录检查
- 完整扫描结构（包括所有子目录）
- 必读 P0/P1 文件
- 检查每个子目录的关键文件
- 根据目标选择性深入 P2/P3/P4
- 执行交叉验证

### 大型项目 (> 100 个文件)
**策略**：目标导向 + 范围明确
- 完整扫描结构
- 必读根目录关键文件
- 聚焦用户问题相关模块
- 深入读取相关模块的完整内容
- **明确声明**哪些内容未读取及原因
